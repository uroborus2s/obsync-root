# PRD

**项目名称：** stratix框架以及生态  
**文档状态：** 草稿  
**负责人：** 仓库维护者  
**主要读者：** 维护者 | 架构 | 开发 | QA  
**上游输入：** 当前状态分析 | 项目章程  
**下游输出：** 需求分析 | 技术选型 | 实施计划  
**关联 ID：** `REQ-001` ~ `REQ-005`, `NFR-001` ~ `NFR-004`  
**最后更新：** 2026-03-28  

## 1. 目标

把历史形成的 Stratix 源码仓库接入软件工厂治理，并以当前真实代码和验证结果为基础，建立可持续维护的事实源。

## 2. 业务目标

- `BIZ-001`：让维护者可以快速理解当前仓库真实状态
- `BIZ-002`：让构建、测试、运行和发布结论可证据化复现
- `BIZ-003`：让文档、代码、测试和发布面保持同步

## 3. 范围

- Stratix 核心与生态包
- 非 workspace 的 CLI 预览样例 `examples/web-admin-preview`
- 根级协作协议、文档、memory 和 workitems

## 4. 功能需求

| ID | 需求 |
|---|---|
| `REQ-001` | 项目必须提供正式 `docs/` 与 `.factory/` 事实源，并与代码同步维护。 |
| `REQ-002` | 项目必须对安装、构建、测试、运行和发布状态给出基于命令验证的结论。 |
| `REQ-003` | 项目必须明确当前公共包、私有应用、职责边界和依赖关系。 |
| `REQ-004` | 项目必须把历史 README 中的瞬时状态迁移到 discovery / memory 层。 |
| `REQ-005` | 项目必须用 `BUG / CR / TASK` 管理当前阶段的主要问题和后续动作。 |

## 5. 非功能需求

| ID | 需求 |
|---|---|
| `NFR-001` | 所有阶段性结论必须能追溯到实际代码、配置、文档或命令结果。 |
| `NFR-002` | 协作文档必须区分稳定规则与瞬时状态，不得混写。 |
| `NFR-003` | 维护者需要在 Node.js 22+、pnpm 10+ 环境下完成主要验证动作。 |
| `NFR-004` | 变更后必须同步代码、文档、测试和 `.factory/`。 |

## 6. 非目标

- 本轮不修复所有历史缺陷
- 本轮不强行发布所有公共包
- 本轮不完整重写各包 README

## 7. 成功判定

- 维护者可以根据 discovery / requirements / solution 文档理解项目
- 第一批高价值问题已形成正式 workitems
- 当前建议阶段与下一步动作清晰
