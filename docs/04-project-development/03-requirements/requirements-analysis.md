# 需求分析

**项目名称：** stratix框架以及生态  
**文档状态：** 草稿  
**负责人：** 仓库维护者  
**主要读者：** 架构 | 开发 | QA | 维护者  
**上游输入：** PRD | 当前状态分析  
**下游输出：** 需求校验 | 技术选型 | 实施计划  
**关联 ID：** `REQ-001` ~ `REQ-005`, `NFR-001` ~ `NFR-004`, `BUG-001` ~ `BUG-003`  
**最后更新：** 2026-03-28  

## 1. 现有实现已经覆盖的能力

- 已具备公开框架与生态包清单
- 已具备单独可用的 CLI 入口
- 已具备单独可验证的私有管理端应用
- 各生态包已有较完整源码与部分包 README

## 2. 当前缺口

| 缺口 | 对应需求 | 影响 |
|---|---|---|
| 缺少正式 `docs/` 与 `.factory/` | `REQ-001` | 协作事实源缺失 |
| 根验证入口与真实可用性不一致 | `REQ-002`, `NFR-001` | 容易误判项目健康度 |
| 发布面分裂为 manifest/tag/registry 三套事实 | `REQ-002`, `REQ-003` | 无法稳定发布或交接 |
| 顶层 README 混入旧状态与瞬时说明 | `REQ-004`, `NFR-002` | 入口文档失真 |
| 缺少正式工作项控制 | `REQ-005` | 后续修复难以有序推进 |

## 3. 关键分析结论

- 当前最优先的问题不是“代码量不够”，而是“事实源与验证链不一致”。
- 根级问题与包级问题需要拆开治理：
  - 根级：脚本入口、安装复现、发布口径
  - 包级：`@stratix/core` build/test 回归
- 管理后台与 CLI 已证明仓库并非整体瘫痪，因此修复应聚焦在阻塞面，而不是全面推倒。

## 4. 风险展开

| 风险 | 触发信号 | 建议工作项 |
|---|---|---|
| 根测试策略不可信 | `pnpm test` 先被 “No test files found” 击穿 | `BUG-004` |
| 核心包测试回归 | 升级后 core 套件仍需独立复核 | `BUG-002` |
| 安装不可重复 | `frozen-lockfile`、offline install 均失败 | `BUG-003` |
| 版本对外口径混乱 | npm 与 git tag 无法对应本地版本 | `CR-001` |

## 5. 推荐推进顺序

1. 先修根 `test/install` 与 `@stratix/core` 测试阻塞。
2. 再统一 release surface 与 README / docs 口径。
3. 之后进入 design / implementation 阶段，补完整体验证与 CI。
