# 测试计划

**项目名称：** stratix框架以及生态  
**文档状态：** 草稿  
**负责人：** 仓库维护者  
**主要读者：** QA | 开发 | 管理者  
**上游输入：** 当前状态分析 | PRD | 实施计划  
**下游输出：** 测试报告 | 发布结论  
**关联 ID：** `TC-001` ~ `TC-004`, `BUG-001` ~ `BUG-003`  
**最后更新：** 2026-03-28  

## 1. 测试目标

- 先建立“仓级真实验证链”，再追求覆盖率提升。
- 区分根入口验证、包级验证、运行入口验证。

## 2. 测试策略

| 层次 | 目标 | 责任方 | 自动化方式 |
|---|---|---|---|
| 安装验证 | 锁文件、依赖恢复、环境可重复性 | Dev | `pnpm install` 变体验证 |
| 构建验证 | 根脚本与包级脚本一致性 | Dev | `pnpm build`, `pnpm build:all`, targeted builds |
| 单元/集成测试 | 包级稳定性 | Dev / QA | `pnpm test`, targeted package tests |
| 运行入口验证 | CLI 与前端启动可用性 | Dev | `stratix --help`, admin preview |

## 3. 测试入口与出口

### 入口条件

- 依赖已安装
- 当前 workitems 已明确

### 出口条件

- 根 install/build/test 结论一致
- `@stratix/core` build/test 恢复
- 关键运行入口可复现

## 4. 重点覆盖项

| 需求/风险 | 关键场景 | 验证方式 |
|---|---|---|
| `REQ-002` | 根 install/build/test 是否可信 | 仓级命令验证 |
| `REQ-003` | CLI 与管理端入口是否真实可用 | targeted build/test/run |
| `NFR-001` | 结论是否可追溯到命令结果 | 写入 discovery 与 memory |
| `RISK-004` | core 层是否阻塞整体验证 | `@stratix/core` build/test |

## 5. 测试环境与数据

- 环境说明：Node.js 22.x、pnpm 10.x、本地 workspace
- 账号/权限：运行 preview 时需要本地端口监听权限
- 测试数据：以仓库现有测试和 mock 数据为主

## 6. 角色与安排

| 角色 | 负责人 | 职责 |
|---|---|---|
| QA | 待定 | 定义验证矩阵与退出条件 |
| Dev | 维护者 | 修复根脚本与 core 回归 |

## 7. 风险与应对

| 风险 | 影响 | 预案 |
|---|---|---|
| 只看根脚本不看包级结果 | 误判健康度 | 保留仓级和包级双视角 |
| 只看本地 manifest 不看 registry/tag | 误判发布状态 | 发布验证强制三方对照 |
