# @stratix/database 1.1.0 仓储模式说明

## 定位

`@stratix/database@1.1.0` 将 `BaseRepository` 视为应用侧数据库访问的唯一公共入口。

- 单表持久化依然留在表仓储中。
- 多表业务单元应实现为业务仓储。
- service 层负责用例编排和外部适配器调用。
- service 层不应该跨多个仓储手动拼接 SQL。

## 多表业务单元

当一个业务模型横跨多张表时，应该为这个业务单元创建专门的 repository，
并让它继承 `BaseRepository`。

推荐命名示例：

- `WorkflowExecutionRepository`
- `CalendarParticipantUnitRepository`
- `DataScopeUnitRepository`

这类仓储适合承载：

- 跨表读取
- 跨表写入
- 需要事务一致性的状态迁移
- 属于同一个业务能力的原生 SQL

service 层应聚焦于：

- 请求校验
- 调用外部 API
- 将 repository 的结果组合成应用输出

## 长流程工作流

不要在执行长时间外部工作时一直保持数据库事务开启。

推荐模式：

1. 开一个短事务并抢占工作。
2. 持久化快照、版本号、锁令牌或 checkpoint。
3. 在事务外执行慢步骤。
4. 再开一个短事务收口成功或失败。

这样可以在不长时间持有锁的前提下保护数据库一致性。

在具体 repository 类内部，优先使用这些受保护原语来实现可恢复的状态迁移：

- `compareAndSetWhere(...)`
- `compareAndSet(...)`
- `claimById(...)`
- `heartbeatById(...)`
- `saveCheckpointById(...)`

这些原语可以让业务仓储实现所有权校验、乐观锁保护、租约续期和 checkpoint
持久化，而不需要把工作流专用抽象直接塞进 `BaseRepository`。

对于使用 `@stratix/cli@1.1.0` 初始化的新数据库项目，可以使用：

- `stratix generate business-repository <name>`

生成的 repository 会直接示范：

- 多表业务单元仓储
- `claimById(...)`
- `saveCheckpointById(...)`
- 带版本守卫的 `compareAndSet(...)`
- finalize 阶段的 outbox 写入

## 推荐使用的 BaseRepository 能力

新仓储建议优先使用这些能力：

- `tx(...)`
- `txBatch(...)`
- `txWithRetry(...)`
- `query(...)`
- `command(...)`
- `compareAndSetWhere(...)`
- `compareAndSet(...)`
- `claimById(...)`
- `heartbeatById(...)`
- `saveCheckpointById(...)`
- `executeSql(...)`
- `rawQuery(...)`
- `db(...)`
- `reader(...)`
- `writer(...)`

## 时间戳与创建回读

`BaseRepository` 在 1.1.0 中对 `created_at` / `updated_at` 的处理遵循下面约定：

- 只有当仓储 schema 已显式声明这些字段，并且字段类型为字符串兼容类型时，才会自动写入时间值。
- 如果数据库中的真实表结构与仓储内存中的 `tableSchema` 不一致，框架会优先采信数据库 live schema。
- 如果调用方已经传入 `created_at` / `updated_at`，仓储会保留调用方的值。
- 如果字段是 `TIMESTAMP`、`DATETIME` 等非字符串类型，仓储不会自动塞入字符串时间值。
- 框架不再自动往 schema 里补 `created_at` / `updated_at` 字段。

创建记录后的回读也按数据库类型区分：

- PostgreSQL / SQLite：使用 `RETURNING`
- MSSQL：使用 `OUTPUT INSERTED`
- MySQL：先插入，再按主键回读

## 事务经验法则

只有当纯内存计算足够快，并且确实属于一个很短的关键区间时，才应该放进事务。
外部 IO、代码执行、RPC、文件操作和长循环都应该放在事务外，再通过状态迁移、
租约和幂等的 finalize 步骤来保护一致性。
